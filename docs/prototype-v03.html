PatternL13–20

ULID primary keys (String(26), generate_ulid) across all three new tables. Existing tables (service) use auto-increment ints — this is a deliberate new convention for the flow engine.

NoteL21–26

Self-referential FK with use_alter=True — needed because scheduling_flow and scheduling_flow_step reference each other circularly. The named constraint lets Alembic drop it explicitly during migrations.

QuestionL43–45

Each step's typed config lives in a single JSON column. How is the per-step_type schema enforced — on write in Python, or via DB constraint? Trace to the service layer.

QuestionL63–66

unique=True on from_step_id means one transition per source step — a hard linearity guarantee at the schema level. This matches the scope doc decision ("transiciones lineales sólo"). When scope 2 lands (conditional transitions), this constraint will need to be dropped.

Entry point · External (webhook)

Calendly webhook — link booking to run

When the student finishes booking in the Calendly embed, Calendly fires invitee.created. The handler creates the session as it always has, then — if the booking came from a flow — extracts the run id from UTM/prefill, links the session to the run, and advances the run to confirmation.

Per scope: "el webhook de Calendly se extiende mínimamente". Full webhook decomposition is scope 19 (stretch).
