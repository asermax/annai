The watcher (proactive exit detection)

tachikoma/process_supervisor/watcher.py
Bridge Implements R15 — the most interesting part of this PR. Subsystem-owned background loop. Detects exits, stamps the record, then dispatches a Notification event on the shared event bus (cross-feature reference: DLT-098). Priority is Urgent on abnormal exit, Normal on clean exit. Sole producer of exit notifications — stop_process intentionally does not dispatch one.
@@ process_supervisor/watcher.py — new file @@ + async def watch_running(): + while not shutting_down: + for rec in await db.list_running(): + if not _is_alive(rec.pid, rec.pid_start_time): R12: liveness checks both PID and start_time to defeat PID reuse. + exit_code = _try_reap(rec.pid) + await db.mark_exited(rec.id, exit_code) + priority = Urgent if (exit_code != 0 or exit_code is None) else Normal R15: "unknown" exit_code → Urgent (P10 capture mechanism did not yield). + await event_bus.dispatch(Notification(..., priority=priority)) + await asyncio.sleep(POLL_INTERVAL)

Review prompts

  1. R12 PID-reuse guard: verify pid_start_time is read from the OS at spawn time (/proc/{pid}/stat on Linux), not approximated from started_at. Drift between these would defeat the guard.
  2. R15 watcher proactivity: is exit detection truly proactive (background loop runs without external trigger), or does it secretly piggyback on the next tool call? Read the lifecycle of watch_running in bootstrap.py.
  3. R15 sole-producer invariant: grep for Notification( dispatches outside the watcher — there should be zero for exit events. Confirm stop_process does not dispatch one.
  4. R17 (concurrency limit is OUT): is there any accidental cap that would limit concurrent spawns? (file descriptor count, DB connection pool, asyncio semaphore left in by mistake).
  5. R10 log persistence: log files remain on disk after exit — confirm no cleanup code in mark_exited or the watcher.
  6. R14 preamble: the base system prompt should mention these tools. Find where, and check the wording is generic enough that it won't drift from reality when tools change.

Review prompts

  1. Linear-transitions-only invariant: grep scheduling_flow_step_transition for any condition column / JSON field. Scope explicitly defers the matcher DSL — make sure no scaffolding for it leaked in.
  2. Default sequential: is it enforced in the migration (with a backfill of existing rows), or only at the SQLAlchemy default? Existing rows in prod must come out as sequential.
  3. No scheduled job for freeze: grep the jobs dir (app/jobs/) for anything referencing freeze or 10-minute thresholds. The diff already touches app/jobs/ heavily — make sure that's unrelated.
  4. Run-id propagation: trace the round-trip — which endpoint embeds the run id into the Calendly URL, where does the webhook extract it back, and what happens if it's missing/malformed?
  5. Single current_step_id on run: design doc evolves to a richer scheduling_flow_run_step_entry table. Confirm the foundation's simpler shape is a deliberate decision (not yet needed) vs. an oversight that will require a destructive migration.
  6. No cancel API: scope says deferred. Grep the new endpoints for any cancel stub or unused method — if present, it's a foundation-leak from scope 10.
  7. Legacy split is clean: imports in non-legacy code should not reference legacy_* modules, and vice versa. If there's cross-import, the split is leaky.

Design notes from this prototype

What this exercise reveals about the shape

  • Context cards must be typed. "Spec / Design / Plan / Transcript / State / Related / External doc" — the type drives both the icon/treatment and the way annotations reference them (e.g. spec is queryable by requirement ID, design is queryable by section).
  • Bridge notes are the highest-value element. They're where the surface earns its keep — converting "raw diff" into "diff in context". They should be short, opinionated, and link back to context cards by reference (R3, scope-doc-decision, etc.).
  • Reading order is the second most important thing. Both PRs make sense in non-alphabetical order: Tachikoma starts at the MCP registry (the public surface), then drills down through the call stack. Filadd starts at scheduling_mode (the bifurcation), then introduces the new tables, then the API. Alphabetical file lists are useless for review.
  • "No PR body" is a first-class case. The Filadd surface is generated almost entirely from external context — Notion + local files. The surface degrades gracefully and the context rail is where this becomes visible (it's still populated, just from elsewhere).
  • Phase artifacts (Katachi) deserve their own treatment. The fact that spec/design/plan live on the PR temporarily and get deleted on land is meaningful — the surface should reconstruct them from history rather than treating them as "missing".
  • Review prompts are scope-aware. The best prompts are the ones that test invariants the docs commit to ("scope says linear-only — confirm nothing else snuck in"). Generic "is the test coverage good?" prompts add no value.

Open questions surfaced

  • Inline annotations inside the diff (the // annot lines) compete visually with the bridge note above. Both, or one? My guess: bridge note for the "why this stop exists", inline annot for "this specific line matters because…".
  • Should "Next" pointers be linear (one path) or should the surface allow branches (e.g. "the watcher leads to the event bus; the spawn leads to the DB")? Branches are more faithful but harder to render.
  • How does the surface stay in sync when the PR is updated? Re-generate, or stream diff updates into existing stops?
  • For the Filadd case, the state.md file is 60k tokens. Indexing and summarizing it is its own subproblem. Same for transcripts. This is where composability with other skills becomes valuable — a "summarize transcripts" skill produces a context card, the review surface consumes it.
  • The composition model: is each context source fetched by a plugin (Notion plugin, transcript plugin, GitHub PR plugin) that returns a typed card? That's the cleanest abstraction I can see — it answers the "easily composable" requirement directly.

What I'd want to try next

  • Sketch the generator side: given a PR + list of context sources, what's the algorithm that produces this surface? Reading-path inference is the hardest part.
  • Try a third PR with a very different shape — maybe a security/perf fix where the "why" is a single sentence — to stress-test "graceful degradation".
  • Decide on inline annotations vs. side margin annotations (right-rail comments) — different ergonomics for long diffs.
