PatternL13–20

ULID primary keys (String(26), generate_ulid) across all three new tables. Existing tables (service) use auto-increment ints — this is a deliberate new convention for the flow engine.

NoteL21–26

Self-referential FK with use_alter=True — needed because scheduling_flow and scheduling_flow_step reference each other circularly. The named constraint lets Alembic drop it explicitly during migrations.

QuestionL43–45

Each step's typed config lives in a single JSON column. How is the per-step_type schema enforced — on write in Python, or via DB constraint? Trace to the service layer.

QuestionL63–66

unique=True on from_step_id means one transition per source step — a hard linearity guarantee at the schema level. When scope 2 lands (conditional transitions), this constraint will need to be dropped.

Entry point · External (webhook)

Calendly webhook — link booking to run

When the student finishes booking in the Calendly embed, Calendly fires invitee.created. The handler creates the session as before, then — if the booking came from a flow — extracts the run id from UTM/prefill, links the session to the run, and advances the run to confirmation. Per scope: "el webhook de Calendly se extiende mínimamente".

Notes from v04

What changed from v03

  • Top nav now carries the PR-level review actions: Approve and Comment (N). The N badge counts drafts queued during the session.
  • Floating outline nav on the right — compact dashes by default (Notion-minimap style), expands to a labeled outline on hover.
  • Line-hover "+" affordance restored — small button appears on row hover. Click opens an inline editor with two tabs: Ask agent (private, scoped to this line) or Comment for PR (queues a draft).
  • "Ask agent" on side notes is hover-only now — appears when hovering the note (or when expanded). Removes visual noise.
  • Suggestion blocks → "Draft comments". The framing changed: these are not in-page code patches, they are PR comments. Accepting adds them to the comment queue (counter increments in top nav). The optional suggested-change inside is rendered as a labeled block that would become a GitHub ```suggestion snippet in the actual comment.
  • Expand-context UI standardized: every expandable gap now uses the same arrow trio in the gutter — ↑ (50 up), ↕ (full), ↓ (50 down) — with disabled states at file boundaries. Used at top, between hunks, and at bottom.

Things to try in the page

  • Hover the right edge of the page to expand the outline nav.
  • Hover any diff line to see the "+" affordance; click it to open the inline editor (Ask agent / Comment for PR tabs).
  • "Add to drafts" on any of the yellow draft-comment blocks — the header counter increments and the block turns green ("Queued").

Open shape questions for v05+

  • Where does the comment queue live? Clicking "Comment (N)" should probably open a panel listing the queued drafts with edit/remove/submit-all.
  • Agent-asked questions: are they ephemeral (closed when you move on) or persistent (sit on the line as a side-note for later)? The line-hover editor's "Ask agent" tab needs a destination model.
  • Outline nav: should it scroll-track (highlight current section as the page scrolls)? With many groups this becomes important.
  • Tachikoma is still pending as a second example. Its background-watcher entries will be a good stress test of the shape.
