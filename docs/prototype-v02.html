Entry point · HTTP

Advance the run — POST /scheduling-flow-runs/{run_id}/advance/

The student picks an option (or just confirms an intro screen) and the BFF calls this. The engine reads the run's current step, evaluates the (linear, in this slice) transition, and moves the run forward.

The transition model from Base context is what makes this endpoint trivial in this slice — unique=True on from_step_id means there's exactly one outgoing edge per step. Same diff shown again below, narrower, as reminder.

Entry point · External (webhook)

Calendly webhook — link booking to run

When the student finishes booking in the Calendly embed, Calendly fires invitee.created. The webhook handler creates the session as it always has, then — if the booking came from a flow — looks up the run via a prefill/UTM-embedded run id, links the session to the run, and advances it to the confirmation step.

From the scope doc: "el webhook de Calendly se extiende mínimamente". The full webhook decomposition is scope 19 (stretch).

Coexistence layer

The legacy_* split

Existing services (everyone, today) still need the old recurrence-pointer scheduling path. Rather than keep both paths in the same files behind branches, the old code is lifted into parallel legacy_event.py, legacy_service.py, legacy_session.py modules. This lets the new code evolve without breaking the old. Not tech debt — it's a deliberate coexistence strategy.

The split must be clean: non-legacy modules should not import from legacy_*, and vice versa. Cross-imports are the failure mode.

Observations from grounding in real diffs
